## 二阶段提交协议（2PC）TWO Phase Commit

#### **定义:**

为了使基于分布式架构下的所有节点在进行事务处理的过程中能够保持原子性和一致性而设计的一种算法

#### **两个阶段：**

- 第一阶段：事务的准备阶段
  - 事务管理器TM通知资源管理器RM准备分支事务，记录事务日志，并告知事务管理器的准备结果
- 第二阶段：事务的提交或回滚阶段
  - 如果所有的资源管理器在准备阶段都明确返回成功，则事务管理器TM向所有资源管理器RM发起事务提交指令完成数据变更，反之，如果任何一个资源管理器明确返回失败，则事务管理器TM会向所有的资源管理器RM发送事务回滚指令，

#### 缺点：

- 同步阻塞：所有参与者的事务都是阻塞型的，多个分支事务之间是串行的，对于任何指令都必须有明确响应才能继续下一步，否则会处于阻塞状态，暂用的资源会被一致锁定
- 过于保守：任何一个节点失败都会导致数据回滚
- 事务协调者单点故障
- 脑裂问题

## 三阶段提交协议（3PC）Three Phase Commit

二阶段提交协议的改进版本，利用超时机制解决同步阻塞的问题



- CanCommit(询问阶段)：
  - 事务协调者向参与者发送事务执行请求，询问是否可以完成指令，参与者只需要回答是或者不是即可，不需要做真正的事务操作，这个阶段会有超时终止机制
- PreCommit(准备阶段):
  - 事务协调者会根据参与者的反馈结果决定是否继续执行，如果在询问阶段所有参与者都返回可以执行操作，则事务协调者会向所有参与者发送PreCommit请求，参与者收到请求后，写redo和undo日志，执行事务操作但不提交事务，然后返回ACK响应等待事务协调者的下一步通知，如果在询问阶段任意参与者返回不能执行操作的结果，那么事务协调者会向所有参与者发送事务中断请求。
- DoCommit(提交或回滚阶段):
  - 这个阶段会存在两种结果，任然根据上一步骤的执行结果来决定DoCommit的执行方式。如果每个参与者在PreCommit阶段都返回成功，那么事务协调者会向所有参与者发起事务提交指令，反之，如果参与者中任一参与者返回失败，那么事务协调者就会发起终止指令来回滚事务

一旦超时，在三阶段提交协议先任然可能出现数据不一致的情况

## 二阶段提交的实现

### X/Open分布式事务模型

X/Open DTP 是X/Open这个组织定义的一套分布式事务标准。这个标准提出使用两阶段提交来保证分布式事务的完整性

#### 三大角色：

AP：Application 表示应用程序

RM：Resource Manager，表示资源管理器

TM：Transaction Manager，

- 表示事务管理器，事务的协调者，负责协调和管理事务，提供AP编程接口，或管理RM
- TM能够管理多个数据库事务
- TM是一个全局事务管理器，它可以管理多个资源管理器的事务，TM最终会根据各个分支事务的执行结果进行提交或者回滚，如果注册的所有分支事务中任何一个节点事务执行失败，TM会触发各个RM的事务回滚操作。

#### 原理：

1. 配置TM，把多个RM注册到TM，相当于TM注册RM作为数据源
2. AP从TM管理的RM中获取连接。
3. AP向TM发起全局事务，生成全局事务ID（XID），XID会通知各个RM。
4. AP通过第二步获得的连接直接操作RM完成数据操作。这时AP在每次操作时会把XID传递给RM
5. AP结束全局事务，TM会通知各个RM全局事务结束
6. 根据各个RM事务的执行结果，进行提交或者回滚

TM和RM之间的事务控制基于XA协议（X/Open提出的分布式事务处理规范）。实现了XA协议的数据库，可以作为RM



### TCC事务补偿方案

在分布式事务场景中每个服务都要实现TCC。

把一个完整的业务拆分成三个步骤：



Try：对数据的校验或者资源的预留

Confirm：真正执行的任务，只操作try阶段预留的资源

Cancel：取消执行，释放Try阶段预留的资源
